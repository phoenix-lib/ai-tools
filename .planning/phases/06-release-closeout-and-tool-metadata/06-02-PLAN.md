# Plan 06-02: Centralize Tool Metadata and Packet Constants

## Gate Resolution

- **Research Gate:** user selected `Research first`; evidence in
  `06-RESEARCH.md`.
- **Upstream Freshness Gate:** passed before planning at
  `2079ab9626e9f9ed256512091f9c5ea473582885`.
- **AI Tools Self-Use Gate:** skip during this plan's implementation unless
  metadata changes alter generated packet semantics enough to require a
  focused auditor self-run. Full self-use evidence is planned in `06-03`.
- **New Tool Intake Gate:** not applicable. This plan adds shared metadata for
  the existing auditor and packet standard, not a new tool registry.
- **Cross-Repo Outgoing Need Gate:** not applicable; metadata stays AI
  Tools-owned and preserves kit-compatible packet fields.
- **Project Changelog Gate:** applies because packet metadata source and
  downstream compatibility evidence change.

<objective>
Create one code-owned source for review packet constants and
`contract-drift-auditor` metadata so package version, schema version, artifact
names, tool name, and policy hash sources cannot drift silently.
</objective>

<context>
- Phase context: `.planning/phases/06-release-closeout-and-tool-metadata/06-CONTEXT.md`
- Research: `.planning/phases/06-release-closeout-and-tool-metadata/06-RESEARCH.md`
- Current renderer: `shared/review-packet-renderer.js`
- Current auditor manifest: `tools/contract-drift-auditor/index.js`
- Packet schemas: `standards/review-packet/schemas/`
- Existing renderer tests: `test/shared/review-packet-renderer.test.js`
</context>

<must_haves>
- META-01: tool metadata and protocol versions are centralized.
- D-08: add a single code-owned metadata source, likely `shared/tool-metadata.js`.
- D-09: `package.json` remains the package version source.
- D-10: auditor runner, renderer, schema/packet tests, and examples should consume shared metadata where practical.
- D-15: `06-02` may precede fixture work because examples should use final metadata values.
- Do not build the Phase 9 full tool registry here.
- Do not change packet schema semantics unless required for centralization.
- Keep generated artifact names exactly `REVIEW-SUMMARY.json`, `EVIDENCE.json`, `FINDINGS.md`, `RECOMMENDED-ACTIONS.md`.
</must_haves>

<threat_model>
- **T-06-02-01 Registry creep (medium):** Metadata helper could become the Phase 9 tool registry.
  - **Mitigation:** Limit helper to constants and package version lookup.
  - **Verification:** no maturity/activation/owner registry fields added.
- **T-06-02-02 Packet compatibility drift (high):** Refactor could rename artifacts or schema versions.
  - **Mitigation:** tests assert exact artifact names and schema version.
  - **Verification:** schema-output and renderer tests pass.
- **T-06-02-03 Broken package version lookup (medium):** Moving version lookup could emit `0.0.0` accidentally.
  - **Mitigation:** test helper reads `package.json` from repo root.
  - **Verification:** metadata unit test compares package version to `package.json`.
</threat_model>

  `secret_policy`;

  `0.0.0` on unreadable metadata.

version from `package.json`, and falls back deterministically for an invalid
root path.
