# Plan 14-01: Ledger Schema Contracts and Fixture Expectations

## Gate Resolution

- **Research Gate:** user selected research-first; evidence is in
  `14-RESEARCH.md`.
- **Upstream Freshness Gate:** resolved in `14-RESEARCH.md`; upstream
  `ai-workspace-kit` was clean and already up to date at `683afc7`.
- **AI Tools Self-Use Gate:** deferred to `14-02`, because this plan defines
  schemas before generated ledger output can be validated against them.
- **New Tool Intake Gate:** skipped with reason. This plan creates standards
  for existing `project-context-ledger` artifacts, not a new tool.
- **Cross-Repo Gates:** skipped with reason. No upstream mutation or external
  project planning-state copy occurs.

<objective>
Create the public JSON Schema contract for the six project-context-ledger
artifact families, document the schema set, and add focused tests that compile
the schemas and prove the intended strictness with small fixture expectations.
</objective>

<must_haves>

  refs, confidence, source metadata, category, and last checked timestamp.
- LEDGER-SCHEMA-02: define schemas for `COMMANDS.json`, `CONTRACTS.json`,
  `SKILLS.json`, and `DECISIONS.json` with stable IDs, evidence refs,
  confidence, and source metadata.
- LEDGER-SCHEMA-03: define a schema for `CACHE-MANIFEST.json` with tool
  metadata, schema version, source hashes, generated artifact lists, and policy
  metadata.
- D-01, D-02, D-03, D-04: place schemas under
  `standards/project-context-ledger/schemas/`, use
  `https://ai-tools.local/schemas/project-context-ledger/` IDs, factor common
  definitions only when useful, and document the schema set.
- D-05, D-06, D-07, D-08: preserve record-artifact top-level arrays, keep
  `CACHE-MANIFEST.json` as an object, keep review packet schema versioning in
  review packet metadata, and use `project-context-ledger/v1`.
- D-09, D-10, D-11, D-12, D-13: require record IDs, confidence, evidence refs,
  source metadata, strict properties, and test-only invariants for uniqueness
  and evidence joins.
- D-14, D-15, D-16, D-17, D-18, D-19: encode the artifact-specific required
  fields for facts, commands, contracts, skills, decisions, and cache manifest.
- D-20, D-21, D-22: preserve current cache-manifest policy metadata and model
  `scanned_sources` plus both `previous_manifest` states strictly.
</must_haves>

<threat_model>
- **T-14-01-01 Breaking existing consumers (high):** schemas could wrap array
  artifacts in a new object envelope.
  - **Mitigation:** top-level arrays remain the contract for facts, commands,
    contracts, skills, and decisions.
  - **Verification:** schema contract tests validate top-level arrays and
    reject wrapper-only examples.
- **T-14-01-02 Over-loose schemas (high):** accepting arbitrary properties
  would make downstream consumers depend on accidental prose fields.
  - **Mitigation:** use `additionalProperties: false` on schemas and nested
    objects.
  - **Verification:** invalid examples with unknown properties fail.
- **T-14-01-03 Encoding cross-artifact invariants in JSON Schema (medium):**
  uniqueness and evidence joins are brittle in schema-only form.
  - **Mitigation:** keep those checks in tests, not in complex schema tricks.
  - **Verification:** this plan documents the invariant tests required by
    `14-02`.
- **T-14-01-04 Cache manifest over-design (medium):** schema work could turn
  into Phase 16 diff/scope semantics.
  - **Mitigation:** validate only current manifest states and metadata.
  - **Verification:** no `scope`, `changed_facts`, disposition, or triage
    fields are added by this plan.
</threat_model>

<action>
Define strict schemas for the five top-level record arrays:

1. Use draft 2020-12 and schema IDs under
   `https://ai-tools.local/schemas/project-context-ledger/`.
2. Reuse common definitions for confidence, record IDs, evidence ref IDs,
   source path, source hash, and timestamp-like strings where helpful.
3. `FACTS.json`: require `id`, `category`, `confidence`, `evidence_refs`,
   `source_path`, `last_checked`, and at least one of `text` or `value`.
4. `COMMANDS.json`: require `id`, `kind`, `command`, `confidence`,
   `evidence_refs`, and `source_path`; keep `name`, `package_path`, and
   `source_sha256` optional.
5. `CONTRACTS.json`: require `id`, `type`, `path`, `confidence`,
   `evidence_refs`, and `source_path`, matching the preferred backfill in
   `14-02`.
6. `SKILLS.json`: require `id`, `name`, `path`, `confidence`, and
   `evidence_refs`; keep `source_sha256` optional.
7. `DECISIONS.json`: require `id`, `type`, `text`, `confidence`,
   `evidence_refs`, and `source_path`; keep structured `value` optional.
8. Set `additionalProperties: false` at every level controlled by these
   schemas.
</action>

<action>
Define the strict cache manifest schema:

1. Require `schema_version`, `run_timestamp`, `tool`, `ledger_artifacts`,
   `packet_artifacts`, `scanned_sources`, `policy_hashes`,
   `ignored_generated_packet_dirs`, `path_only_secret_paths`, and
   `previous_manifest`.
2. Require `schema_version` to be `project-context-ledger/v1`.
3. Validate `tool` as an object containing at least tool name and version
   fields currently generated by the ledger.
4. Validate `ledger_artifacts` against the six ledger artifact names.
5. Validate `packet_artifacts` against the four standard packet artifact names.
6. Require each `scanned_sources` record to have `path` and `path_only`; when
   `path_only` is false, require a SHA-256 hash; when true, forbid the hash.
7. Validate `policy_hashes`, `ignored_generated_packet_dirs`, and
   `path_only_secret_paths` as deterministic metadata arrays/objects matching
   current generator output.
8. Allow exactly two `previous_manifest` states:
   `{ compared: false, reason }` and
   `{ compared: true, changed_sources }`, with changed entries requiring
   `path`, `previous_sha256`, and `current_sha256`.
</action>
