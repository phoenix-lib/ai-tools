# Plan 10-02: Gates Scan CLI, Packet Output, and Self-Use Validation

## Gate Resolution

- **Research Gate:** user selected Research first; evidence in
  `10-RESEARCH.md`.
- **Upstream Freshness Gate:** resolved in `10-UPSTREAM-UPDATE-REVIEW.md`.
  Upstream confirms gate linting belongs to external AI Tools evidence and
  must not be executed or required by the kit.
- **AI Tools Self-Use Gate:** applies. After implementation, run `gates-scan`
  against this repo as evidence. Also run `contract-drift-auditor` because
  package metadata, docs, registry, and changelog change.
- **New Tool Intake Gate:** resolved. Promote `gates-scan` from planned to
  implemented or validated only after code, tests, docs, and self-use evidence
  make that true.
- **Cross-Repo Outgoing Need Gate:** skipped with reason. The plan does not
  need kit-owned implementation; it preserves external-tool boundaries.
- **Project Changelog Gate:** applies. Record CLI, validation, upstream impact,
  compatibility impact, and migration notes.

<must_haves>
- GATELINT-01: user can run an evidence-only mechanical gate linter reporting
  missing gate blocks, duplicate gate IDs, stale paths, missing source layers,
  missing changelog entries, conflicting required/forbidden wording,
  unresolved references, and gates without observable artifact output.
- D-03: use `gates-scan --project <path> --out <dir>` and emit
  `REVIEW-SUMMARY.json`, `EVIDENCE.json`, `FINDINGS.md`, and
  `RECOMMENDED-ACTIONS.md`.
- D-04: findings are evidence only and must not approve, reject, revise, or
  auto-create gate work.
- D-05: implement deterministic checks first; avoid broad prose semantics.
- D-06: fixture-driven tests prove broken cases; real repo self-use is
  verification evidence only.
- D-07: update registry maturity only when implementation and validation are
  true.
- Keep default shell behavior non-breaking; do not add automatic fail policy in
  this MVP unless implemented as opt-in with tests.
</must_haves>

<threat_model>
- **T-10-02-01 Target mutation (critical):** scanner could write inside the
  audited project.
  - **Mitigation:** require `--out`, use `assertSafeOutputDir`, and test unsafe
    output rejection before packet writes.
  - **Verification:** CLI/integration tests assert output inside project is
    rejected and no packet appears there.
- **T-10-02-02 Semantic overreach (high):** findings could imply automatic gate
  approval/rejection.
  - **Mitigation:** finding titles, README, and recommended actions use
    "review evidence" wording; no action mutates planning state.
  - **Verification:** docs/tests assert evidence-only wording and rejected
    mutating flags.
- **T-10-02-03 False positive flood on historical artifacts (high):** scanner
  could flag old phases that predate gates.
  - **Mitigation:** completed phase checks should focus on artifacts with
    explicit applicable gate cues or current milestone/governance docs; legacy
    handling should be info or skipped with evidence.
  - **Verification:** compatible fixture and real self-use interpretation.
- **T-10-02-04 Packet count drift (medium):** JSON, Markdown, and CLI status
  could diverge.
  - **Mitigation:** use shared packet renderer and `deriveCounts`.
  - **Verification:** schema/packet tests read emitted artifacts.
- **T-10-02-05 Registry truth drift (medium):** registry could mark validated
  without self-use evidence.
  - **Mitigation:** update to `validated` only after focused tests, full tests,
    and real self-use run pass; otherwise use `implemented`.
  - **Verification:** registry test plus summary evidence.
</threat_model>
