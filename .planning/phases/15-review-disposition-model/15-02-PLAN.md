# Plan 15-02: Rollup Disposition Consumption and Review Surface

## Gate Resolution

- **Research Gate:** user selected research-first; evidence is in
  `15-RESEARCH.md`.
- **Dependency Gate:** depends on `15-01` schema and fingerprint helper. Do
  not invent a second fingerprint algorithm in rollup.
- **AI Tools Self-Use Gate:** applies. After synthetic tests pass, run
  `review-packet-rollup` with explicit disposition input over real AI Tools
  packet outputs and external `--out`; record the output as evidence only.
- **Project Changelog Gate:** applies. Update `CHANGELOG.md` with changed
  scope, validation, compatibility, and upstream impact.
- **Cross-Repo Gates:** skipped with reason. This plan does not mutate
  `ai-workspace-kit` and does not make AI Tools a runtime dependency there.

<objective>
Extend `review-packet-rollup` so it can optionally consume explicit
`REVIEW-DISPOSITIONS.json` files, join them to normalized packet findings via
stable fingerprints, and emit separate disposition review artifacts while
preserving the original review packet findings, status, severity, evidence,
and counts.
</objective>

<must_haves>
- DISP-01: rollup must not delete, rewrite, or suppress original source
  findings when dispositions are provided.
- DISP-02: disposition files consumed by rollup must validate against the
  Phase 15 schema.
- DISP-03: expired or stale dispositions remain visible as review-required
  context.
- DISP-04: joins are separate metadata and preserve source severity, status
  contribution, evidence refs, blockers, required decisions, and status
  derivation.
- DISP-05: joins use `shared/finding-fingerprint.js` and survive
  occurrence-normalized rollup finding ids.
- D-23 through D-28: first consumer is `review-packet-rollup`, add narrow
  optional `--dispositions <file...>`, emit `DISPOSITION-INDEX.json`, and do
  not modify source findings or standard `generated_artifacts`.
- D-29 through D-32: add a bounded mechanical review surface only if cheap; it
  must not rank business priority or declare safe-to-ignore.
- D-35 through D-38: add join, expiry visibility, non-mutation, and self-use
  evidence while keeping synthetic fixtures as the oracle.
- D-39 through D-42: no Phase 16 scope/diff, no Phase 17 shared CLI migration,
  no portfolio scanning, and no automatic disposition generation.
</must_haves>

<threat_model>
- **T-15-02-01 Hidden suppression (high):** active dispositions could
  accidentally remove findings or alter status derivation.
  - **Mitigation:** compute disposition index after normalization and packet
    summary derivation; leave `normalized.findings` untouched.
  - **Verification:** tests compare summary counts, finding ids, severity,
    status contribution, evidence refs, and status with and without
    dispositions.
- **T-15-02-02 Expired context invisible (high):** expired dispositions could
  be silently ignored.
  - **Mitigation:** `DISPOSITION-INDEX.json` has explicit `expired` records
    and counts; add a visible rollup-owned finding only if needed for top-level
    review status.
  - **Verification:** expired fixture produces a visible expired bucket and
    review-required evidence.
- **T-15-02-03 Fingerprint mismatch (high):** rollup could compute
  fingerprints from different fields than the standard helper.
  - **Mitigation:** all joins call `shared/finding-fingerprint.js`.
  - **Verification:** occurrence-normalized duplicate-id fixture still matches
    by fingerprint.
- **T-15-02-04 Input mutation (high):** disposition consumption could write
  into packet directories or disposition file locations.
  - **Mitigation:** all generated artifacts are written only under `--out`.
  - **Verification:** tree-hash tests cover packet input dirs and fixture
    disposition dirs.
- **T-15-02-05 CLI scope creep (medium):** plan could drift into Phase 17
  shared CLI behavior.
  - **Mitigation:** only add `--dispositions`; leave format, quiet, fail-on,
    and shared parser work untouched.
  - **Verification:** CLI tests assert help and parser behavior only for the
    new option and existing review-only rejected flags.
- **T-15-02-06 Dashboard semantics (medium):** bounded surface could become
  priority ranking.
  - **Mitigation:** dashboard, if added, only reports mechanical counts/top
    groups and links to full index.
  - **Verification:** no safe-to-ignore, priority, gate, merge, or roadmap
    fields appear in dashboard output.
</threat_model>

<action>
Implement disposition loading and indexing:

1. Load zero or more disposition files from explicit paths.
2. Validate each file against `REVIEW-DISPOSITIONS.schema.json` with AJV.
3. Convert invalid or stale schema/tool-version input into visible index
   entries, not thrown-away metadata.
4. For each normalized finding, compute a stable fingerprint using source
   tool, source check id, source path, normalized target, and optional source
   packet id.
5. Join disposition records by `finding_fingerprint`.
6. Build `DISPOSITION-INDEX.json` with deterministic sections for matched,
   unmatched, expired, stale, invalid, and findings without disposition.
7. Include counts and enough refs to trace back to finding ids, source packet
   ids, disposition ids, evidence refs, and source paths.
8. Do not mutate `normalized.findings` or source packet records.
</action>

- Unmatched dispositions are present in an unmatched bucket.
- Invalid or stale-version disposition files remain visible.
- Findings without disposition are counted and listed deterministically.
</acceptance_criteria>
