# Plan 15-01: Review Disposition Schema and Finding Fingerprints

## Gate Resolution

- **Research Gate:** user selected research-first; evidence is in
  `15-RESEARCH.md`.
- **Dependency Gate:** depends on Phase 14 schema discipline and Phase 13
  rollup normalization behavior. This plan creates the standard and helper
  before consumers use them.
- **AI Tools Self-Use Gate:** deferred to `15-02`, because this plan defines
  the artifact contract before rollup can consume real disposition files.
- **New Tool Intake Gate:** skipped with reason. This is a standard/helper
  plan for existing packet consumers, not a new auditor.
- **Cross-Repo Gates:** skipped with reason. No upstream checkout mutation or
  cross-repo request handling is in scope.

<objective>
Define the public `REVIEW-DISPOSITIONS.json` schema, document review
dispositions as human metadata rather than suppression, and add a deterministic
finding fingerprint helper that later packet consumers can use for stable
joins.
</objective>

<must_haves>
- DISP-01: dispositions are sidecar human review metadata and never delete or
  rewrite source findings.
- DISP-02: schema requires finding id, stable fingerprint, source tool, source
  path, source check id, reason, owner, reviewed timestamp, expiry timestamp,
  evidence refs, tool version, and schema version.
- DISP-05: fingerprint helper derives IDs from stable evidence fields so
  dispositions survive occurrence-normalized rollup finding IDs.
- D-01 through D-06: place the standard under
  `standards/review-disposition/`, use `review-disposition/v1`, define
  `REVIEW-DISPOSITIONS.json` as a top-level object, and keep
  `FINDING.schema.json` unchanged.
- D-07 through D-12: require `id` and `finding_fingerprint`, keep exact
  `finding_id`, add `shared/finding-fingerprint.js`, and avoid mutable prose,
  severity, status, and occurrence suffixes in fingerprints.
- D-13 through D-22: encode required fields, optional provenance, required
  expiry, and the stored status enum.
- D-33, D-34: add examples or in-test fixtures for active, expired, unmatched,
  and stale-version records, with AJV valid/invalid coverage.
</must_haves>

<threat_model>
- **T-15-01-01 Suppression semantics (high):** docs or schema names could imply
  that a disposition hides a finding.
  - **Mitigation:** README explicitly states dispositions do not modify source
    findings, status, severity, counts, evidence refs, blockers, required
    decisions, or gate outcomes.
  - **Verification:** schema contract tests ensure dispositions are separate
    artifacts; no review packet finding schema changes occur.
- **T-15-01-02 Fingerprint instability (high):** fingerprints could include
  occurrence-normalized IDs, title, summary, severity, or status.
  - **Mitigation:** helper accepts a stable input object and documents excluded
    fields.
  - **Verification:** tests prove fingerprints stay equal when finding id
    occurrence suffix, title, summary, severity, and status contribution
    change.
- **T-15-01-03 Hidden indefinite review state (high):** dispositions without
  expiry could become stale permanent suppressions.
  - **Mitigation:** `expires_at` is required and date-time validated.
  - **Verification:** invalid fixture without `expires_at` fails schema.
- **T-15-01-04 Over-loose artifact contract (medium):** accepting arbitrary
  fields would create accidental consumer dependencies.
  - **Mitigation:** use `additionalProperties: false` for controlled objects.
  - **Verification:** invalid fixture with an unknown field fails schema.
- **T-15-01-05 Over-scoped helper (medium):** helper could read files or target
  projects.
  - **Mitigation:** fingerprint helper is pure and receives already-normalized
    values.
  - **Verification:** tests call helper directly with objects only.
</threat_model>

<action>
Create the public review disposition standard:

1. Document `REVIEW-DISPOSITIONS.json` as a human-authored sidecar artifact.
2. Set schema version to `review-disposition/v1`.
3. Use a top-level object with `schema_version` and `dispositions`.
4. Allow optional source/review metadata at the top level where useful, but do
   not require packet co-location.
5. Define each disposition record with the Phase 15 required fields.
6. Use status enum:
   `accepted_current_issue`, `accepted_historical_noise`, `not_actionable`,
   `false_positive`, and `needs_followup`.
7. Require `expires_at`; do not add stored `expired` status.
8. Use strict schemas with `additionalProperties: false`.
9. State that disposition files may live beside rollup output or in an
   external review directory, while source packet directories remain read-only.
</action>

<action>
Add the deterministic fingerprint helper:

1. Export a function such as `createFindingFingerprint(input)`.
2. Normalize source tool, source check id, source path, target, and optional
   source packet id into a canonical object.
3. Convert Windows separators to `/`, reject or normalize absolute path forms
   into the documented stable representation, and use `unknown` for missing
   source paths.
4. Hash the canonical sorted object with SHA-256 and return `fp.<hash>`.
5. Keep the helper pure: no filesystem reads, no target project reads, no
   packet loading, and no date/time dependency.
6. Export any small normalization helpers only if tests or rollup consumption
   need them.
</action>
<verify>
`npm.cmd test -- test/review-disposition/finding-fingerprint.test.js`
</verify>
<acceptance_criteria>
- Same stable fields always produce the same fingerprint.
- Backslash and slash path variants produce the same fingerprint.
- Missing source path consistently uses `unknown`.
- Title, summary, severity, status contribution, and occurrence-normalized
  finding id changes do not affect the fingerprint.
- Changing source tool, source check id, source path, target, or source packet
  id changes the fingerprint.
</acceptance_criteria>
</task>
