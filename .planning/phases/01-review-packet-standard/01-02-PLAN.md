# Plan 01-02: Define Evidence, Actions, and Tool Manifest Metadata

<objective>
Complete the focused schema set by defining evidence refs, recommended actions, and tool manifest metadata, then wire those schemas into the review summary and finding schemas created by plan 01-01.
</objective>

<context>
- Depends on plan 01-01 outputs in `standards/review-packet/`.
- Phase context decisions D-06 through D-11 and D-19 through D-30 are mandatory.
- `ai-workspace-kit` manifest concepts should be mirrored conceptually, not cloned as the packet format.
</context>

<must_haves>
- D-04: `EVIDENCE.json` is the machine evidence index and must avoid long copied source content.
- D-09: `EVIDENCE-REF.schema.json` owns evidence identity and proof.
- D-10: `RECOMMENDED-ACTION.schema.json` owns action guidance, not patches.
- D-11: `TOOL-MANIFEST.schema.json` owns tool identity, version, schema versions, input args, safety profile, read/write behavior, policy hashes, and run timestamp.
- D-19: Evidence paths are normalized relative paths using `/`, not absolute local paths.
- D-20: Readable evidence uses `sha256` content hashes, while structured packet hashing follows canonical JSON rules.
- D-21: Secret-like path-only evidence must use `path_only: true` and must not require or encourage content hashes.
- D-22: Evidence refs include optional `line`; ranges may be supported only if cheap.
- D-23: Evidence refs include a short `reason`.
- D-24: Unknown or stale facts must be explicit in evidence/finding data.
- D-25: Evidence snippets are not part of V1.
- D-26: Mirror `ai-workspace-kit` concepts, not its exact manifest.
- D-27: Packet-compatible concepts include source commit, tool version, generated files, policy hash, requested outputs, shared review summary, evidence files, blockers, required decisions, rejected assumptions, and preserved stricter local rules.
- D-28: `ai-workspace-kit` remains a reference and optional integration target, not a runtime dependency.
- D-29: Machine values use underscore-safe names where possible.
- D-30: Adapter-specific outputs remain out of scope for Phase 1.
</must_haves>

<tasks>
<task id="T1" type="create">
<files>
- `standards/review-packet/schemas/EVIDENCE-REF.schema.json`
</files>
<action>
Create a strict draft 2020-12 schema for evidence refs. It must require:
- `id`
- `path`
- `evidence_type`
- `reason`
- `confidence`
- `path_only`

It must allow optional `sha256`, `line`, and `range` fields. Add schema rules so non-path-only readable evidence can carry `sha256`, while `path_only: true` is valid without `sha256`. Document in `description` fields that secret-like files must use path-only evidence and V1 evidence snippets are not supported.
</action>
<verify>
PowerShell:
`node -e "JSON.parse(require('fs').readFileSync('standards/review-packet/schemas/EVIDENCE-REF.schema.json','utf8')); console.log('evidence schema json ok')"`
</verify>
<acceptance_criteria>
The evidence schema validates normalized, reasoned, confidence-marked evidence refs and supports secret-safe path-only entries.
</acceptance_criteria>
</task>

<task id="T2" type="create">
<files>
- `standards/review-packet/schemas/RECOMMENDED-ACTION.schema.json`
</files>
<action>
Create a strict draft 2020-12 schema for recommended actions. It must require:
- `id`
- `summary`
- `rationale`
- `target_owner`
- `human_review_required`

It may allow optional `suggested_command`, `suggested_file`, and `finding_refs`. It must not include patch bodies, patch bundles, or automatic apply instructions. Add descriptions that recommended actions are guidance for humans or future tools, not mutation instructions.
</action>

PowerShell:
`node -e "JSON.parse(require('fs').readFileSync('standards/review-packet/schemas/RECOMMENDED-ACTION.schema.json','utf8')); console.log('action schema json ok')"`
</verify>
<acceptance_criteria>
The action schema separates next-step guidance from findings and from patch/application behavior.
</acceptance_criteria>
</task>

<task id="T3" type="create">
<files>
- `standards/review-packet/schemas/TOOL-MANIFEST.schema.json`
</files>
<action>
Create a strict draft 2020-12 schema for tool manifest metadata. It must require:
- `tool_name`
- `tool_version`
- `schema_versions`
- `input`
- `generated_files`
- `run_timestamp`
- `safety_profile`
- `policy_hashes`

It must support optional `source_commit`, `requested_outputs`, `read_write_behavior`, and `target_project`. Model concepts from `ai-workspace-kit` such as generated files, policy hash, requested outputs, source commit, and run timestamp, but do not reuse the exact adoption manifest status strings or adapter-specific fields.
</action>

</verify>
<acceptance_criteria>
The manifest schema captures V1 packet metadata needed by later auditors without making `ai-workspace-kit` a runtime dependency.
</acceptance_criteria>
</task>
