# Plan 01-01: Define Review Packet Schemas and Taxonomy

<objective>
Create the Phase 1 review packet foundation: the standards directory, top-level packet guidance, packet status/severity/confidence taxonomy, and initial strict schemas for `REVIEW-SUMMARY.json` and finding objects.
</objective>

<context>
- Phase context: `.planning/phases/01-review-packet-standard/01-CONTEXT.md`
- Research: `.planning/phases/01-review-packet-standard/01-RESEARCH.md`
- Seed: `ai-review-packet-standard/README.md`
- Guide: `AI-AGENT-IMPLEMENTATION-GUIDE.md`
- Reference concepts: `.external/ai-workspace-kit/ADAPTER-GENERATION.md`
</context>

<must_haves>
- D-01: V1 mandatory packet artifacts are `REVIEW-SUMMARY.json`, `FINDINGS.md`, `EVIDENCE.json`, and `RECOMMENDED-ACTIONS.md`.
- D-02: `REVIEW-SUMMARY.json` is documented as the top-level machine entrypoint.
- D-03: Markdown artifacts are documented as projections rendered from JSON data.
- D-05: `RISK-MATRIX.md`, `RECOMMENDED-PATCHES.md`, patch bundles, and adapter-specific outputs remain out of scope.
- D-06: Use focused schemas rather than one monolithic packet schema.
- D-07: `REVIEW-SUMMARY.schema.json` owns packet-level fields.
- D-08: `FINDING.schema.json` owns finding-level fields.
- D-12: V1 packet statuses are exactly `pass`, `info`, `human_review_required`, and `blocked`.
- D-13: Status remains packet-level outcome; severity remains finding-level impact.
- D-14: V1 finding severities are exactly `critical`, `high`, `medium`, `low`, and `info`.
- D-15: V1 confidence values are exactly `verified`, `inferred`, `unknown`, and `stale`.
- D-16: `blocked` is reserved for packet-safety blockers.
- D-17: `human_review_required` is reserved for evidence-backed human decisions.
- D-18: Shared summary/count rendering is documented as mandatory to prevent JSON/Markdown/CLI drift.
</must_haves>

<task id="T2" type="create">
<files>
- `standards/review-packet/schemas/REVIEW-SUMMARY.schema.json`
</files>
<action>
Create a draft 2020-12 JSON Schema for `REVIEW-SUMMARY.json` with strict `additionalProperties: false`. It must require:
- `schema_version`
- `status`
- `counts`
- `tool`
- `target_project`
- `generated_artifacts`
- `findings`
- `evidence_file`
- `recommended_actions_file`
- `blockers`
- `required_decisions`
- `rejected_assumptions`
- `preserved_stricter_local_rules`

The schema must define status enum values exactly as the locked Phase 1 decision states. Counts must include total finding count, severity counts, blocker count, required decision count, rejected assumption count, and preserved stricter local rule count. The schema may reference `FINDING.schema.json` and `TOOL-MANIFEST.schema.json` if the local validator in plan 01-03 can resolve those refs.
</action>
<verify>
PowerShell:
`node -e "JSON.parse(require('fs').readFileSync('standards/review-packet/schemas/REVIEW-SUMMARY.schema.json','utf8')); console.log('summary schema json ok')"`
</verify>
<acceptance_criteria>
The schema parses as JSON, is strict, requires packet-level machine fields, and uses the exact V1 status enum.
</acceptance_criteria>
</task>

<task id="T3" type="create">
<files>
- `standards/review-packet/schemas/FINDING.schema.json`
</files>
<action>
Create a draft 2020-12 JSON Schema for finding objects with strict `additionalProperties: false`. It must require:
- `id`
- `severity`
- `title`
- `summary`
- `confidence`
- `status_contribution`
- `source_check_id`
- `evidence_refs`
- `recommended_action_refs`

Severity and confidence enums must exactly match Phase 1 context. `status_contribution` must use the packet status enum but remain a contribution field, not the packet outcome. Evidence refs and recommended action refs may initially be arrays of ids or schema refs, but the shape must be compatible with plan 01-02 refinement.
</action>
<verify>
PowerShell:
`node -e "JSON.parse(require('fs').readFileSync('standards/review-packet/schemas/FINDING.schema.json','utf8')); console.log('finding schema json ok')"`
</verify>
<acceptance_criteria>
The schema parses as JSON, validates the finding taxonomy, and preserves the separation between packet status and finding severity.
</acceptance_criteria>
</task>
