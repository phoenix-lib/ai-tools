# Plan 13-01: Rollup Fixtures, Packet Validation, Provenance, and Grouping Model

## Gate Resolution

- **Research Gate:** user selected Research first; evidence in `13-RESEARCH.md`.
- **New Tool Intake and Placement Gate:** resolved in `13-CONTEXT.md`. The
  destination is `tools/review-packet-rollup/`; the capability is a packet
  consumer, not a domain auditor.
- **AI Tools Self-Use Gate:** applies later in Phase 13 after the runnable CLI
  exists. This plan builds synthetic fixtures and pure packet mechanics first.
- **Upstream Freshness Gate:** already resolved for v2.1 roadmap planning.
  Phase 13 does not depend on new `ai-workspace-kit` runtime behavior.
- **Cross-Repo Incoming/Outgoing Gates:** skipped with reason. This plan does
  not add kit-owned adoption/bootstrap behavior or mutate the upstream checkout.

more existing review packet directories, validate `REVIEW-SUMMARY.json` and

before wiring the user-facing CLI.

<must_haves>
- ROLLUP-01: consume two or more packet directories without mutating packet
  sources or target projects.
- ROLLUP-02: validate every input `REVIEW-SUMMARY.json` and each
  `EVIDENCE.json` entry before aggregation.
- ROLLUP-03: compute groups by source tool, status, severity,
  `source_check_id`, `status_contribution`, and source path.
- ROLLUP-04: preserve blockers, required decisions, and
  `human_review_required` contributors as separately inspectable data.
- ROLLUP-05: produce provenance model data for packet input paths, artifact
  hashes, source tool names, tool versions, schema versions, and future
  `PACKET-INDEX.json` / `ROLLUP-GROUPS.json`.
- ROLLUP-06: do not suppress source findings, mark findings safe-to-ignore, or
  convert evidence into automatic gate, roadmap, merge, or disposition
  decisions.
- D-07, D-08, D-09: required machine inputs are `REVIEW-SUMMARY.json` and
  `EVIDENCE.json`; validate the summary schema and each evidence ref entry.
- D-10, D-11, D-12: Markdown packet projections are not required; invalid
  packet JSON creates blocked rollup findings while valid packets still index.
- D-13, D-14, D-15, D-16, D-17: preserve source findings with deterministic
  prefixed ids and refs, keep source severity/confidence/check/status fields,
  and store provenance outside finding objects.
- D-18, D-19, D-20, D-21, D-22: define model data that can later render the
  standard packet plus `PACKET-INDEX.json` and `ROLLUP-GROUPS.json`, sorted
  deterministically and without semantic labels.
- D-23, D-24, D-25: derive source path groups from validated evidence refs,
  allow a finding in multiple source path groups, and use deterministic
  `unknown` buckets for unresolved evidence refs.
- D-26, D-27, D-28: derive status and counts mechanically from normalized
  findings and copied decision arrays, not manual prose summaries.
- D-31, D-33, D-34, D-35: use canonical deterministic JSON expectations and
  synthetic fixtures as the primary oracle for invalid packets, duplicate ids,
  blockers, required decisions, and multiple evidence refs.
</must_haves>

<threat_model>
- **T-13-01-01 Schema-invalid rollup findings (high):** provenance fields
  could be added directly to findings even though the finding schema forbids
  additional properties.
  - **Mitigation:** normalize finding ids and refs only; store provenance in
    packet index and group records.
  - **Verification:** schema fixture tests validate generated finding shapes.
- **T-13-01-02 Trusting invalid packets (high):** malformed source packets
  could be silently skipped, producing an apparently clean aggregate.
  - **Mitigation:** every invalid source packet gets an index record and a
    blocked rollup finding.
  - **Verification:** invalid fixture asserts blocked status and valid packets
    still appear in groups.
- **T-13-01-03 Path grouping from prose (medium):** parsing file paths from
  finding summaries could create false source path groups.
  - **Mitigation:** derive source paths only from validated evidence refs.
  - **Verification:** fixture with misleading summary prose groups by evidence
    path only.
- **T-13-01-04 Non-deterministic outputs (medium):** group ordering or packet
  ids could change between runs.
  - **Mitigation:** deterministic packet ids, stable sorting, fixed-clock tests,
    and canonical JSON.
  - **Verification:** same fixture run produces identical JSON models.
- **T-13-01-05 Semantic creep (medium):** grouping fields could become
  priority/disposition/safe-ignore labels.
  - **Mitigation:** group only the mechanical dimensions listed by ROLLUP-03.
  - **Verification:** tests and docs reject `safe_to_ignore`, `priority`, and
    disposition fields in group output.
</threat_model>
