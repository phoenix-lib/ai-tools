# Plan 02-01: Implement Shared Safety Helper Foundation

<objective>
Create the reusable `shared/` helper foundation for Phase 2: canonical JSON,
output path isolation, secret-like path policy, default ignore policy, and
deterministic target file walking.
</objective>

<must_haves>
- D-01: Phase 2 is owned by AI Tools safety primitives, not adoption/bootstrap behavior.
- D-02: Do not add `ai-workspace-kit` as a runtime dependency.
- D-03: Adapt principles from upstream helpers into independent generic helpers.
- D-04: Do not build adapter generation, adoption packets, merge routing, or contract installers.
- D-05: Shared boundary work is limited to packet-compatible metadata and evidence refs.
- D-06: Report-generating tools require explicit output dirs outside the target.
- D-07: Resolve existing paths through real paths and handle missing output suffixes.
- D-08: Path containment comparisons must be Windows-safe.
- D-09: Reject unsafe output paths before creating output directories.
- D-10: Do not add a broad output-guard bypass.
- D-11: Output guard errors must explain the rejected path.
- D-12: Secret-like paths are path-only evidence by default.
- D-13: Classify `.env`, `.env.*`, secret/token/credential names, `.key`, `.pem`, `.p12`, `.pfx`, `id_rsa`, `id_dsa`, and `id_ed25519` as secret-like.
- D-14: Secret evidence refs use normalized relative paths, `secret_path`, `path_only: true`, reason, and verified confidence when the path exists.
- D-15: Secret path-only evidence must not include `sha256`, snippets, or content-derived facts.
- D-16: Structure-only env examples are deferred; Phase 2 default remains conservative.
- D-17: Tests must prove secret sentinels never appear in generated outputs or report text.
- D-18: Ignore policy excludes workspace noise, generated packets, nested checkouts, and fixture expected/output trees.
- D-19: Default ignored dirs include `.git`, `node_modules`, `.pnpm-store`, `vendor`, `dist`, `build`, `.next`, `out`, `coverage`, `.tmp`, and `tmp`.
- D-20: Generated packet dirs are ignored by marker files, including AI Tools and ai-workspace-kit markers.
- D-21: Nested `ai-workspace-kit` checkouts are ignored by package name or source layer markers.
- D-22: Fixture expected/output trees are ignored unless explicitly targeted.
- D-23: File walker returns deterministic sorted relative paths using `/`.
- D-24: Secret classification happens before any content read.
- D-25: File walker supports an explicit deterministic file cap.
- D-32: Reusable mechanics belong in `shared/`.
- D-33: Use CommonJS modules and `node:test`.
- D-34: Promote canonical JSON into `shared/canonical-json.js`.
- D-35: Expected helpers include `path-guard`, `secret-policy`, `ignore-policy`, `file-walker`, `tree-hash`, and `canonical-json`.
- D-36: Shared helpers expose small deterministic functions, not CLI behavior.
- D-37: Tests live under `test/shared/`.
- D-38: Upstream freshness gate ran before planning.
- D-39: Latest reviewed upstream update reaches `48ec037d058747151c320ced9c0ee1e1d247d4b1`.
- D-40: Use upstream review-summary/tooling lessons as compatible input only.
- D-41: Future phase plans record upstream commits in the upstream review note.
</must_haves>

<threat_model>
- **T-02-01-01 Unsafe output containment bypass (high):** A string-prefix path check could allow sibling paths or fail on Windows case differences.
  - **Mitigation:** Implement path containment with `path.resolve`, real existing ancestors, `path.relative`, and case-normalized comparison on Windows.
  - **Verification:** `test/shared/path-guard.test.js` covers equal target, child output, sibling output, nonexistent output dirs, and Windows-like case normalization where applicable.
- **T-02-01-02 Secret content exposure (critical):** A helper could hash or read `.env`/key material before classifying it.
  - **Mitigation:** Keep `secret-policy` path-based and make evidence creation path-only.
  - **Verification:** `test/shared/secret-policy.test.js` asserts no `sha256` or content fields for secret evidence refs.
- **T-02-01-03 Evidence pollution (medium):** Generated packets or nested checkouts could be treated as target evidence.
  - **Mitigation:** Use marker-based generated packet detection and nested checkout detection in `ignore-policy`.
  - **Verification:** `test/shared/file-walker.test.js` covers ignored dirs, generated packet marker dirs, nested kit checkout markers, and fixture expected/output trees.
</threat_model>

<task id="T2" type="create">
<files>
- `shared/path-guard.js`
- `test/shared/path-guard.test.js`
</files>
<action>
Implement output path guard helpers:
- `realOrResolved(filePath)` resolves existing paths with native realpath and
  preserves missing suffixes.
- `isInsideOrEqual(parentDir, candidateDir)` returns true only when the candidate
  equals or is inside the parent.
- `assertSafeOutputDir(targetDir, outDir)` throws before writes if output equals
  or is inside target.

Keep error text actionable: name the rejected target/output paths and tell the
caller to use a separate review directory outside the target.
</action>
<verify>
PowerShell:
`npm.cmd test -- test/shared/path-guard.test.js`
</verify>
<acceptance_criteria>
Unsafe output paths are rejected deterministically before directory creation,
while sibling output directories remain allowed.
</acceptance_criteria>
</task>

<task id="T3" type="create">
<files>
- `shared/secret-policy.js`
- `test/shared/secret-policy.test.js`
</files>
<action>
Implement secret path classification and path-only evidence creation:
- `normalizeEvidencePath(relPath)` converts backslashes to `/` and rejects
  absolute paths for evidence use.
- `isSecretLikePath(relPath)` classifies default secret-like names from D-13.
- `secretEvidenceRef({ id, path, reason })` returns an
  `EVIDENCE-REF.schema.json` compatible object with `evidence_type:
  "secret_path"`, `path_only: true`, and `confidence: "verified"`.

The module must not read file contents or compute content hashes.
</action>
<verify>

</verify>
<acceptance_criteria>

</acceptance_criteria>
</task>
