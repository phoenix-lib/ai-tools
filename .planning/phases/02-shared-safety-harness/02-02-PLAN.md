# Plan 02-02: Build Fixture Harness and Tree Hash Mutation Proof

<objective>
Create the raw target tree hashing helper and reusable Phase 2 target fixtures
needed to prove future read-only commands do not mutate audited projects.
</objective>

<context>
- Depends on plan 02-01 shared helper foundation.
- Phase context decisions D-26 through D-31 define fixture and mutation proof behavior.
- Research validation requires fixture coverage for all TEST-07 scenarios.
</context>

<must_haves>
- D-26: Fixtures cover clean project, mature project, stale source layer, missing command, secret-like files, mixed package managers, and generated packet inside target tree.
- D-27: Fixture targets live under `test/fixtures/targets/<scenario>/input/`.
- D-28: Tree hashing uses a raw recursive file walk over the target tree, not the audit ignore policy.
- D-29: Tree hash uses sorted normalized relative file paths plus raw file bytes with NUL separators and SHA-256.
- D-30: No-mutation tests hash before and after read-only operations and fail on file additions, deletions, renames, or content changes.
- D-31: Fixture outputs used by tests live outside the target tree and are cleaned by test helpers.
- D-17: Secret sentinel fixture values must support later non-leakage tests.
- D-22: Fixture expected/output trees remain ignored unless directly targeted.
</must_haves>

<threat_model>
- **T-02-02-01 Mutation hidden by ignore rules (high):** A tool could write into `node_modules`, `.git`, temp, or generated packet dirs and evade an audit-walk-based hash.
  - **Mitigation:** `shared/tree-hash.js` must walk all files under the target input tree without using `ignore-policy`.
  - **Verification:** `test/shared/tree-hash.test.js` proves ignored-dir content changes alter the hash.
- **T-02-02-02 Fixture leakage into target inputs (medium):** Expected output files inside `input/` could make no-mutation checks noisy or invalid.
  - **Mitigation:** Keep expected/output material outside target inputs; add helper assertions for fixture shape.
  - **Verification:** `test/shared/fixture-helpers.js` and `test/shared/safety-harness.test.js` assert fixture inputs are clean.
- **T-02-02-03 Secret sentinel accidental exposure (critical):** Secret fixtures need realistic sentinels but must not leak into outputs.
  - **Mitigation:** Place sentinel values only in secret-like files and use later tests to scan generated output.
  - **Verification:** Plan 02-03 runs output scanning against `SECRET_SENTINEL_DO_NOT_LEAK` strings.
</threat_model>

<tasks>
<task id="T1" type="create">
<files>
- `shared/tree-hash.js`
- `test/shared/tree-hash.test.js`
</files>
<action>
Implement raw tree hashing:
- `walkTreeFiles(dir)` recursively returns all file paths under `dir`, sorted by
  normalized relative path.
- `relativeTreeFiles(dir)` returns normalized `/` paths.
- `treeHash(dir)` creates a SHA-256 digest by updating with relative path,
  NUL, raw file bytes, NUL for every file.

Tests must prove stable hashes for unchanged trees, changed hashes for content
edits, and changed hashes for files under directories the audit walker would
ignore.
</action>
<verify>
PowerShell:
`npm.cmd test -- test/shared/tree-hash.test.js`
</verify>
<acceptance_criteria>
Tree hashing is deterministic and detects any target tree mutation regardless
of audit ignore policy.
</acceptance_criteria>
</task>
